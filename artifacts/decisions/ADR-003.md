# ADR-003: FastAPI Generator 与 SSE 推流结构选用

## 状态：已接受
## 上下文：
在 Sprint 3 中，我们需要将诊断数据以不间断流的形式推送到前端面板上。

## 决策：
在 FastAPI 中采用 `StreamingResponse` 伴随 `AsyncGenerator` 协程推流方案（标准的 Server-Sent Events Header：`text/event-stream`）。放弃 WebSocket 的全双工架构。

## 理由：
1. **单向传导特性**：内省流面板的设计核心是从下往上单向吐血（Agent 执行流的日志只读呈现），不需要浏览器端做高频互动回复。SSE 完美契合并且比 WebSocket 更轻量。
2. **兼容性与重连机制**：浏览器的 `EventSource` 会自带掉线自动重连机制（Retry），并且对现有反向代理更友好（HTTP 协议）。

## 后果：
客户端需要配置并监听符合 W3C 的 `EventSource`。若出现跨域情况 CORS 配置会比纯粹的 JSON API 更敏感一些。需要在前端处理可能的容错回落机制。
