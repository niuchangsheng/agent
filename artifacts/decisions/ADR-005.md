# ADR-005: 设计自动工作流节点下发 ADR

## 状态：已接受
## 上下文：
在 Sprint 5 我们希望达成一个功能：在 Agent 经历完若干次沙箱碰壁后成功解决问题，系统能自动聚合经验。

## 决策：
开发一个端点 `/api/v1/tasks/{task_id}/generate-adr`，扫描带有 `is_success=True` 的执行轨迹（Traces），将其中涉及的实施补丁（`applied_patch`）整合渲染为 Markdown 文档。
在物理存储层设定双写（Dual-write）策略：
1. 以 `Adr` ORM 表记录进 SQLite。
2. 物理下发存盘进项目的 `artifacts/decisions/`。

## 理由：
1. 双写策略：SQLite 保存用于界面接口的日后迅速分页提取，物理 `artifacts/` 下发保存能够直接跟随 Git 提交供后人浏览或离线查询。
2. 将该功能独立为路由端点（避免融合到沙箱运行中），是为了遵守低耦合和单一职责；以后如果接入真正的 LLM （大语言模型），这个汇总调用本身会及其耗时，应当作为异步操作在结尾单独调用。

## 后果：
此项操作由于依赖宿主的文件系统 `os.makedirs` 以及 `open`，在日后如果上云端 Docker K8S 部署时需要规划 Volume 挂载卷的挂载目录才能确保物理文件存留。
