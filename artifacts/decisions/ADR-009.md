# ADR-009: Redis 队列持久化方案选型

**状态**: 已采纳 (2026-04-13)

## 上下文

Sprint 7 实现的内存队列存在以下问题：
1. 进程重启后所有队列任务丢失
2. Worker 崩溃后无法可靠恢复任务
3. 无法支持多进程/分布式部署

Sprint 9 需要实现任务队列的持久化，确保服务重启后未完成任务能够恢复。

## 决策

采用 **Redis Sorted Set + Hash** 作为队列持久化方案。

### 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 持久化存储 | Redis | 高性能、支持 Sorted Set 优先级队列、原子操作 |
| Python 客户端 | redis-py (asyncio) | 官方支持、异步兼容 FastAPI |
| 降级方案 | InMemoryQueue | Redis 不可用时自动降级，保证服务可用 |

### Redis Key 设计

```
seca:queue:pending       # Sorted Set - 待执行任务 (按优先级 + 时间排序)
seca:queue:running       # Hash - 运行中任务详情
seca:queue:task:{id}     # Hash - 任务完整详情
```

### 降级策略

1. Redis 连接失败 → 自动降级到 InMemoryQueue
2. 降级时记录警告日志，不阻塞服务启动
3. Redis 恢复后不自动切换（避免状态不一致）

## 权衡

### 优势 (+)
- Redis 性能优异，支持高并发
- Sorted Set 天然支持优先级队列
- 原子操作保证数据一致性
- 降级策略保证服务可用性

### 劣势 (-)
- 引入 Redis 外部依赖
- 增加运维复杂度
- Redis 单点故障风险（未来可通过 Cluster 解决）

### 备选方案

1. **SQLite 队列**: 无需外部依赖，但并发性能差
2. **RabbitMQ/AMQP**: 功能强大，但过于重量级
3. **PostgreSQL LISTEN/NOTIFY**: 数据库耦合，不推荐

## 影响

### 技术影响
- 新增 `app/queue/` 模块
- `main.py` lifespan 增加 Redis 初始化逻辑
- 测试套件需要支持 Redis skip

### 运维影响
- 部署文档需要增加 Redis 安装步骤
- 环境变量 `REDIS_URL` 配置

### 向后兼容
- API 端点保持不变
- 优先级字段默认值为 0（无优先级）
