---
description: 自动驱动规划、构建、QA 测试闭环直至完成的所有宏运行调度器
---

1. 检测并判断 `artifacts/product_spec.md` 这份基础总纲是否合法存在。缺失则立刻直接先转跑 `/plan` 工作流逻辑体系。
2. 检索并解析 `product_spec.md`，展开对于所有分级 Sprint 列表状态 `[ ]`, `[/]`, `[!]`, `[x]` 标记的排查：
   【大循环开始】
   - 当检测到任务树内存续标志为 `[ ]` (待开始) 或 `[!]` (曾经被 QA 挡回重工) 的 Sprint 项目时：
     A. 调用并完整单趟执行对应的 `/build` 工作流。（Generator 进行修复/新建）。
     B. 当它声明完毕后，紧随其后执行 `/qa` 工作流进行严密打分判断。
   - 【熔断暂停校验】
     如果同一 Sprint 经历了 `/build` -> `/qa` -> 被重打为 `[!]` 的轮次累积连续超过了 **3 次**。意味着模型出现了修复打转或长上下文丢失造成的智力瓶颈。主控程序应当直接：
       1. 终止大循环！
       2. 将阻断错误明细详细反馈写入 `artifacts/handoff.md`
       3. 回绝并在聊天上抛出明确求助："Sprint X 连续三次未能破局通过 QA 标准，请帮我人工审阅或给予提示帮助"。
   - 如果 `/qa` 给出了 `[x]` 放行（并且顺次推到尾）：大循环正常结束！
3. 当且仅当读取发现 `product_spec.md` 所有的 Sprint 行全员晋级为 `[x]` 已完成。不再执行构建，而是跳转进入 `/release` 的结项报告流转中。
4. 确保在全部中断出、完成出的地方通过 `artifacts/handoff.md` 履行最新进度的留存与汇报。
