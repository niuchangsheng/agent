# /run - 永动迭代调度器

## 执行流程

### 0. 状态检测（每次执行前）
执行 `node .claude/hooks/run-status-check.js` 显示当前进度摘要。

### 1. 检测 product_spec.md
若不存在 → 先转跑 `/plan` 工作流。

### 2. 版本状态检测
解析 `product_spec.md` 中的"版本历史"表：
- 若所有版本状态均为 `✅ 已完成` → 执行步骤 3（自动规划新版本）
- 若存在状态为 `🔄 规划中` 或 `⏳ 进行中` 的版本 → 执行步骤 4（继续构建该版本）

### 3. 【永动模式】所有版本已完成，自动生成下一版本规划
A. 调用 `/plan` 工作流，基于当前系统能力和用户潜在需求，规划下一个版本（如 v1.1 → v1.2）。
B. 新版本规划完成后，将版本状态标记为 `🔄 规划中`。
C. **持久化**：执行 `git add -A && git commit -m "feat(plan): 规划新版本 vX.Y"`。
D. 进入步骤 4 继续构建。

### 4. 迭代构建循环
针对当前最新版本中状态为 `[ ]` (待开始) 或 `[!]` (曾被 QA 挡回) 的 Sprint：
- A. 调用 `/build` 工作流执行 TDD 开发。
- B. 紧随其后执行 `/qa` 工作流进行严密打分。
- C. **持久化**：若 Sprint 状态发生变更，执行 `git add -A && git commit -m "chore(sprint N): 状态更新"`。
- 【熔断机制】若同一 Sprint 连续 3 次被 QA 打回为 `[!]`：
  1. 终止循环
  2. 将阻断错误明细写入 `artifacts/handoff.md`
  3. 执行 git 提交留存当前状态
  4. 抛出求助："Sprint X 连续三次未能通过 QA，请人工审阅"

### 5. 版本完成判定
当当前版本的所有 Sprint 均标记为 `[x]`：
- 将该版本状态更新为 `✅ 已完成`
- 执行 git 提交留存版本完成状态
- 若有待进行版本，跳回步骤 4
- 若无待进行版本，跳回步骤 3

### 6. 持久化协议
确保在任何中断点或完成点：
- `artifacts/handoff.md` 已更新最新进度
- 状态变更已 git commit

---

## 状态标记说明
- `✅ 已完成` - 该版本所有 Sprint 已通过 QA
- `🔄 规划中` - 该版本 Sprint 已定义但未开始
- `⏳ 进行中` - 该版本部分 Sprint 正在开发

## 跨会话恢复
- 每个 Sprint 完成后 git commit，确保状态可跨会话恢复
- 再次执行 `/run` 时自动检测当前状态并继续
- git 历史作为状态变迁的权威记录

## 相关命令
- `/run-continue` - 从上次中断处继续执行（详细模式）
