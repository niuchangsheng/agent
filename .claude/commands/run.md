---
description: 自动驱动规划、构建、QA 测试闭环的永动迭代调度器
---

1. 检测并判断 `artifacts/product_spec.md` 是否合法存在。缺失则立刻先转跑 `/plan` 工作流。

2. **版本状态检测**：解析 `product_spec.md` 中的"版本历史"表，找到当前最新版本的 Sprint 列表。
   - 若所有版本的状态均为 `✅ 已完成` → 执行步骤 3（自动规划新版本）
   - 若存在状态为 `🔄 规划中` 或 `⏳ 进行中` 的版本 → 执行步骤 4（继续构建该版本）

3. **【永动模式】所有版本已完成，自动生成下一版本规划**：
   A. 调用 `/plan` 工作流，基于当前系统能力和用户潜在需求，规划下一个版本（如 v1.1 → v1.2）。
   B. 新版本规划完成后，将版本状态标记为 `🔄 规划中`，并立即进入步骤 4 继续构建。
   C. **持久化**：执行 `git add -A && git commit -m "feat(plan): 规划新版本 vX.Y"` 提交状态。

4. **迭代构建循环**：针对当前最新版本中状态为 `[ ]` (待开始) 或 `[!]` (曾被 QA 挡回) 的 Sprint：
   - A. 调用 `/build` 工作流执行 TDD 开发。
   - B. 紧随其后执行 `/qa` 工作流进行严密打分。
   - C. **持久化**：若 Sprint 状态发生变更，执行 `git add -A && git commit -m "chore(sprint N): 状态更新 - [x]/[!]"`。
   - 【熔断机制】若同一 Sprint 连续 3 次被 QA 打回为 `[!]`：
     1. 终止循环，将阻断错误明细写入 `artifacts/handoff.md`
     2. 执行 git 提交留存当前状态
     3. 抛出求助："Sprint X 连续三次未能通过 QA，请人工审阅或给予提示"

5. **版本完成判定**：当当前版本的所有 Sprint 均标记为 `[x]`：
   - 将该版本状态更新为 `✅ 已完成`
   - 执行 git 提交留存版本完成状态
   - 若有 `⏳ 进行中` 的下一版本，跳回步骤 4 继续
   - 若无待进行版本，跳回步骤 3 触发新版本规划

6. **跨会话持久化协议**：
   - 每个 Sprint 完成后必须 git commit，确保状态可跨会话恢复
   - `handoff.md` 必须包含最新更新时间、当前 Sprint 进度、关键决策摘要
   - 会话中断后重新开始时，hook 会自动检测并提示继续

7. 确保在任何中断点或完成点，通过 `artifacts/handoff.md` 留存最新进度。

---
## 状态标记说明
- `✅ 已完成` - 该版本所有 Sprint 已通过 QA
- `🔄 规划中` - 该版本 Sprint 已定义但未开始
- `⏳ 进行中` - 该版本部分 Sprint 正在开发

## 跨会话恢复
- 每次执行 `/run` 前，`SessionStart` hook 会自动检测 `handoff.md` 和 `product_spec.md`
- 若有待进行的 Sprint，会输出提示引导继续执行 `/run`
- git 提交历史作为状态变迁的权威记录
