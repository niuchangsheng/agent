---
description: 执行 Evaluator 角色，作为零容忍的主干级测试与评分审计官
---

1. 读取 `.claude/agents/evaluator.md` 获取"零容忍"要求的安全和体验 QA 规则指南，并牢记"反自我宽容规则"（不能通过浏览代码判定功能、拒绝马虎及主观好评）。
2. 读取 `artifacts/handoff.md` 和 `artifacts/sprint_contract.md` 定位刚刚接受审查的功能底线与实现目标。
3. 【TDD 合规检查】
   - 审计生成器 Generator 写出来的代码仓库行为。审查其测试代码对验收合同具体标准的覆盖比例及实际落地。如果发现缺少足够边界防御，抑或是"先实现了大堆应用代码后随便补充了几个单元测试"，在代码质量纬度毫不留情打≤4 分。

3.5 【冒烟门禁 — BLOCKER 级别】
   - 此步骤 **不可被跳过或推迟**。
   - 启动前端 dev server（如 `npm run dev`）和后端 server（如 `uvicorn`）。
   - 分别使用 `curl` 命令实测两个服务端口均返回 HTTP 200 正常数据流。
   - 若任一服务启动或连通失败：立即终止评分流程，在报告中写入失败原因，并将 Sprint 判定为 `[!]` 打回！

4. 【浏览器驱动及模拟】
   - 可在终端直接调用应用拉起脚手架（如 `npm run dev` / `uvicorn`）使服务变成在线。
   - 要求优先凭借浏览器工具真正模拟人类执行点击、边界测试探测和异常路径（例如无断网测试模拟）。如果是完全 API 后端逻辑，亦必须使用相应的集成请求去轰炸端点。
   - 在这之中顺带进行回归验证，针对前某几个关键功能的关联流派发起回测确认功能未跌档退化。
5. 【结构化评分】
   - 将上述一切表现落实为以下四个维度的具体给分（各 1-10 分）。**不允许在任何维度打 `-` 或跳过**：功能完整性（占比 35%）、设计质量（体验 UI 及样式，模板病占比低为高分，25%）、代码质量（容错处理与强类型，覆盖率，20%）、用户体验（20%）。只有全部项不低于 6 且加权总分≥7.0 方可签批发版。
   - 每个维度的分数必须在报告中附带 ≥1 条可追溯的证据引用（来自终端/curl 响应/截图）。无证据支撑的满分无效。
   - 输出这份具有严格格式并附有指导整改意见的报告至 `artifacts/qa_feedback.md`。
6. 判断总分是否过阈：
   - 达标：把 `artifacts/product_spec.md` 此局 Sprint 状态标注改作 `[x]`。
   - 打回：该状态倒转为 `[!]`。
7. 将此次的结论和遗留要求同步改入 `artifacts/handoff.md`。提示下一级动作循环修复或推进。
